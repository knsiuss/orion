# Production Hardening Design

**Date:** 2026-03-09
**Status:** Approved
**Scope:** Local-first personal AI companion, deployable on laptop / home server / VPS (all platforms)
**Strategy:** Native install as primary path, Docker as VPS/server alternative

---

## 1 — Process Stability (5→9)

### Graceful Shutdown (`src/core/shutdown.ts`)

Register on `SIGTERM` and `SIGINT`. Shutdown sequence (10s timeout, then force exit):

1. `outbox.stopFlushing()` + one final `outbox.flush()`
2. `channelManager.stop()`
3. `sidecarManager.stopAll()`
4. `PRAGMA wal_checkpoint(TRUNCATE)` — clean WAL before disconnect
5. `prisma.$disconnect()`
6. `process.exit(0)`

Wire into `startup.ts`: replace the existing SIGTERM stub with the new handler.

### Auto-start Templates (`scripts/`)

Generated by `pnpm run setup:autostart` — detects OS and prints platform-specific instructions:

| File | Platform |
|------|----------|
| `scripts/edith.service` | systemd (Linux / VPS) |
| `scripts/com.edith.plist` | launchd (macOS) |
| `scripts/edith-task.xml` | Windows Task Scheduler |
| `ecosystem.config.cjs` | PM2 (cross-platform) |

`setup:autostart` script writes the correct absolute paths into the template and prints the install command.

## 2 — Database Durability (7→9)

### Auto-backup (`src/database/backup.ts`)

- Schedule: every 1 hour via daemon
- Before copy: `PRAGMA wal_checkpoint(TRUNCATE)` to ensure consistency
- Destination: `EDITH_BACKUP_DIR` env var, default `.edith/backups/`
- Filename: `edith-YYYY-MM-DD-HH.db`
- Retention: keep last 24 files (24h window), prune older

### Migration on Startup

- Call `prisma migrate deploy` during `initializeServices()`
- On failure: log error + continue (graceful degradation, don't crash EDITH)

## 3 — Deployment (1→7)

### `/health` Endpoint

No auth required. Returns `200` (ok) or `503` (degraded/down).

```json
{
  "status": "ok",
  "uptime": 3600,
  "version": "0.1.0",
  "db": "ok",
  "channels": { "telegram": true, "discord": false },
  "outbox": { "pending": 2, "deadLetters": 0 }
}
```

### Dockerfile (multi-stage)

- Stage 1 `builder`: `node:22-alpine` — pnpm install + build
- Stage 2 `python`: `python:3.11-slim` — pip install requirements
- Stage 3 `runtime`: `node:22-alpine` — merge artifacts, non-root user (`edith:edith`)

### `docker-compose.yml`

```yaml
services:
  edith:
    build: .
    volumes:
      - ./prisma:/app/prisma      # DB persistence
      - ./.edith:/app/.edith      # backups + outbox
      - ./.env:/app/.env:ro       # secrets (read-only)
    ports:
      - "3000:3000"
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      retries: 3
```

## 4 — Security (6→8)

### Environment-Aware Gateway Auth (`src/gateway/middleware/auth.ts`)

```
if gateway bound to 127.0.0.1 → skip auth (trusted local)
else → require: Authorization: Bearer <EDITH_API_TOKEN>
  - missing/wrong token → 401
  - EDITH_API_TOKEN empty + public binding → startup warning + reject all requests
```

New config vars: `EDITH_API_TOKEN`, `GATEWAY_HOST` (default `127.0.0.1`).

### TLS

EDITH does not terminate TLS internally. Document Caddy/Nginx reverse proxy setup in `docs/deployment/reverse-proxy.md` for VPS deployments.

### Audit Log

`AuditRecord` model already in schema (Phase 28). Wire to pipeline:
- Log on every inbound message + outbound send
- Risk level: low / medium / high based on CaMeL classification

## 5 — Observability (7→9)

### Correlation / Trace IDs

- At pipeline entry: generate `traceId = crypto.randomUUID().slice(0, 8)`
- Thread `traceId` through all log calls in pipeline stages
- Enables: `grep traceId .edith/logs/edith-2026-03-09.log` to trace one full request

### Log File Rotation

Controlled by `LOG_TO_FILE` env var (default `false`):
- Write to `.edith/logs/edith-YYYY-MM-DD.log`
- Rotate at midnight
- Retain last 7 days, auto-prune older files

### Basic Alerting

Controlled by `ALERT_USER_ID` env var. Sends message to that userId via active channel when:
- Outbox dead-letters ≥ 5 within 1 hour
- Circuit breaker open on ≥ 2 channels simultaneously
- Error rate > 10/minute

## 6 — Inbound Rate Limiting (7→9) + Access Control (4→7)

### DM Access Policy

`DM_POLICY_MODE` already in config (Phase 28). Wire to pipeline at Stage 1:

| Mode | Behavior |
|------|----------|
| `open` | All users allowed (default) |
| `allowlist` | Only `ALLOWED_USER_IDS` (comma-separated) |
| `blocklist` | Block `BLOCKED_USER_IDS` (comma-separated) |
| `admin-only` | Only `ADMIN_USER_ID` |

New config vars: `ALLOWED_USER_IDS`, `BLOCKED_USER_IDS`.

### Per-User Inbound Rate Limit

Extend existing pipeline rate limiter:
- Current: global 60 req/min
- Add: per-userId token bucket, 30 req/min per user (configurable via `PIPELINE_RATE_LIMIT_USER_PER_MIN`)
- Over limit: drop message + log warn (no 429 response — channels don't support it uniformly)
